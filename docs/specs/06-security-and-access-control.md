# 06 — Security and Access Control

## Purpose

This chapter describes how the server authenticates users, how role-based
authorisation is enforced at URL and view level, how the bridge between
MySQL identity and MongoDB ownership works, and what a production rebuild
must change before the system can be exposed publicly.

It complements the JPA entity layout in
[`05-persistence-and-mappers.md`](05-persistence-and-mappers.md#mysql-schema)
and the user-facing flows in
[`02-functional-requirements.md`](02-functional-requirements.md).

## Roles and hierarchy

```
ROLE_ADMIN  >  ROLE_AUTHOR  >  ROLE_PLAYER
```

The `Role` enum (`security/model/Role.java`) has exactly three values:
`ADMIN`, `AUTHOR`, `PLAYER`. The hierarchy is configured in
`config/SecurityConfig.java`:

```java
@Bean
static RoleHierarchy roleHierarchy() {
    return RoleHierarchyImpl.fromHierarchy("""
            ROLE_ADMIN > ROLE_AUTHOR
            ROLE_AUTHOR > ROLE_PLAYER
            """);
}
```

A `UserData` may hold any subset of the three roles in its `Set<Role> roles`.
The hierarchy bean ensures a user with ROLE_ADMIN passes any
`hasRole("AUTHOR")` or `hasRole("PLAYER")` check without needing those roles
explicitly. View-level `@RolesAllowed("ROLE_PLAYER")` therefore admits authors
and admins as well.

## Spring Security wiring

`SecurityConfig` is annotated `@Configuration @EnableWebSecurity
@EnableMethodSecurity`. Highlights:

| Concern | Configuration |
|---------|---------------|
| Request cache | `NullRequestCache` — Vaadin's `RootView` does the post-login routing, so we never want Spring Security to save and replay a URL. |
| Session management | `maximumSessions(1)`, `maxSessionsPreventsLogin(false)` — one active session per user; logging in again invalidates the previous session. |
| Remember-me | Signed cookie with key from `${security.remember-me.key}`; `alwaysRemember(false)` (the user opts in via the form). |
| Logout | `logoutSuccessUrl("/")`, `permitAll()`. |
| URL guards (before Vaadin) | `/admin/**` requires `ROLE_ADMIN`, `/author/**` requires `ROLE_AUTHOR`, `/player/**` requires `ROLE_PLAYER`, `/public/**` permitted. |
| URL allow-list | `/login`, `/`, `/VAADIN/**`, `/favicon.ico`, `/robots.txt`, `/manifest.webmanifest`, `/sw.js`, `/offline.html`, `/icons/**`, `/images/**`, `/styles/**`, `/frontend/**`. |
| Vaadin integration | `VaadinSecurityConfigurer.vaadin()` — installs the view-access checker, ignores Vaadin internal requests for CSRF, registers the login view. |
| Login form | `loginPage("/login")`; failure handler redirects to `/login?error` and logs the failed username at INFO. |
| Password encoder | `BCryptPasswordEncoder` (default strength 10). |
| Success handler | Intentionally **not** overridden — Vaadin's configurer handles it. Overriding it caused an ordering conflict that bounced users back to `/login`. |

## Authentication flow

```
Browser → GET /protected         → SecurityFilterChain
   401 → redirect to /login
Browser → GET /login              → LoginView (anonymous)
Browser → POST /login (username, password)
   → DaoAuthenticationProvider
       → CustomUserDetailsService.loadUserByUsername(username)
           → UserRepository.findByUsername(username)
       → BCryptPasswordEncoder.matches(raw, hash)
   → success: redirect to /
   → failure: AuthenticationFailureHandler
       → log "Login failed for user '...': ..."
       → redirect /login?error
Browser → GET /
   → RootView.beforeEnter
       → auth == null || anonymous → /login
       → ADMIN  → /admin/dashboard
       → AUTHOR → /author/dashboard
       → else   → /player/library
```

## Identity model

### `UserData`

JPA `@Entity @Table(name = "users")`, implements Spring Security's
`UserDetails`:

| Field | Type / annotations | Notes |
|-------|---------------------|-------|
| `id` | `String` PK, `length=26`, `nullable=false`, `updatable=false` | ULID, generated by `@PrePersist` if absent. |
| `username` | `String` `unique=true`, `nullable=false` | Looked up by `UserRepository.findByUsername`. |
| `password` | `String` `nullable=false` | BCrypt-encoded. |
| `roles` | `Set<Role>` `@ElementCollection(EAGER) @Enumerated(STRING)` | Stored in a side table by JPA. |
| `isEnabled` | `boolean` | Backs `UserDetails.isEnabled()`. |
| `authorities` | derived | `roles → "ROLE_"+name → SimpleGrantedAuthority`. |

`UserDetails` accessors return:

- `isAccountNonExpired()` → `true` (not modelled today)
- `isAccountNonLocked()` → `true` (not modelled today)
- `isCredentialsNonExpired()` → `true` (not modelled today)
- `isEnabled()` → backed by `isEnabled`

A rebuild MAY model the missing flags; today disabling a user is the only
way to lock them out.

### `CustomUserDetailsService`

Single method: `loadUserByUsername(username)`. Throws
`UsernameNotFoundException` on miss. Delegates to `UserRepository`. Logs the
resolved roles at DEBUG so dev troubleshooting shows what the principal is
carrying.

### `UserService`

Top-level user CRUD:

- `createUser(username, rawPassword, roles)` — encodes password with the
  injected `PasswordEncoder`, persists. Username uniqueness is enforced via
  the database `UNIQUE` constraint; on conflict, surface a user-facing
  `Notification`.
- `findByRole(Role)` — backed by `findByRolesContaining`.
- `save(UserData)` — generic update.
- `delete(String id)` — direct delete; **does not** check for referencing
  `AdventureAuthor` / `AdventurePlayer` rows today (see Known gaps).

## Adventure access bridge: `AdventureAccessService`

The service in `server/security/service/AdventureAccessService.java` owns
every cross-store (MongoDB ↔ MySQL) operation that involves an adventure.
Its rules are normative.

### Read access (`canRead(adventureId, user)`)

- ADMIN — always TRUE.
- AUTHOR of the adventure (`AdventureAuthorRepository.findByAdventureId(...).user.id == user.id`) — TRUE.
- Otherwise: TRUE iff `AdventurePlayerRepository.existsByIdAdventureIdAndUser(adventureId, user)`.

### Write access (`canWrite(adventureId, user)`)

- ADMIN — always TRUE.
- AUTHOR of the adventure — TRUE.
- Otherwise FALSE.

### Lifecycle methods

| Method | Behaviour |
|--------|-----------|
| `findAdventureById(id, user)` | Returns `Optional.empty()` if `canRead` is false; otherwise delegates to `AdventureService.findAdventureById`. |
| `saveAdventureData(data, user)` | Throws `AccessDeniedException` if `canWrite` is false; otherwise delegates. |
| `createAdventure(data, author)` | `@Transactional`. Saves the `AdventureData` first (so the ULID id is fixed), then inserts an `AdventureAuthor(adventureId, author)` row. The MongoDB write is *outside* the JPA transaction; on JPA rollback, the document is orphaned. |
| `deleteAdventure(id, user)` | `@Transactional`. Verifies write access. Deletes `AdventureAuthor` and all `AdventurePlayer` rows, then `AdventureService.deleteAdventure(id)` cascades the document tree. |

### Assignment methods (ADMIN-only — *callers must verify the role*)

| Method | Behaviour |
|--------|-----------|
| `assignAuthor(adventureId, newAuthor)` | `@Transactional`. Throws `IllegalStateException` if a row already exists for `adventureId`. |
| `reassignAuthor(adventureId, newAuthor)` | `@Transactional`. Deletes the existing row and inserts the new one. |
| `removeAuthor(adventureId)` | `@Transactional`. |
| `assignPlayer(adventureId, player)` | `@Transactional`. Idempotent — does nothing if the row already exists. |
| `removePlayer(adventureId, player)` | `@Transactional`. |

### Query methods

- `findAuthorForAdventure(id) → Optional<UserData>`.
- `findPlayersForAdventure(id) → List<UserData>`.
- `getAuthorNamesByAdventureId() → Map<adventureId, username>` — handy for the
  admin assignment view.
- `getAdventuresForUser(user)` — filtered list:
  - ADMIN: all adventures from `AdventureService.getAdventures()`.
  - else: union of authored ids (`AdventureAuthorRepository.findByUser`) and
    assigned ids (`AdventurePlayerRepository.findByUser`), looked up via
    `AdventureService.getAdventuresByIds(ids)`.

A user holding multiple roles gets the **union** of accessible adventures.
This is by design: an author who plays as well sees both lists at once.

## Annotation-driven view authorisation

Every routed view belongs to one of four buckets:

| Annotation | Used by |
|------------|---------|
| `@AnonymousAllowed` | `RootView` (`""`), `LoginView` (`/login`), `AboutView` (`/about`). |
| `@PermitAll` | `LogoutView` (`/logout`); `AdventureAppLayout`, `AdventuresMainLayout` (the layout containers permit any authenticated user — protection is on the routed children). |
| `@RolesAllowed("ROLE_ADMIN")` | `AdminDashboardView`, `UserManagementView`, `AdventureAssignmentView`. |
| `@RolesAllowed("ROLE_AUTHOR")` | All `*EditorView` / `*MenuView` for adventure / location / item / direction / command / message / vocabulary, plus `AuthorDashboardView`, `LocationMapView`, `SpecialWordsView`. |
| `@RolesAllowed("ROLE_PLAYER")` | `PlayerLibraryView`. |

The role hierarchy means an ADMIN passes every check transitively; an
AUTHOR passes PLAYER checks; a PLAYER fails AUTHOR/ADMIN checks.

## Default seed: `DataInitializer`

`config/DataInitializer.java` is a `@Component implements CommandLineRunner`:

- On startup, if `userRepository.findByUsername("admin").isEmpty()`, create:
  - `username = "admin"`
  - `password = BCrypt("admin123")` — **TODO**: change in production.
  - `roles = { ADMIN, AUTHOR, PLAYER }`
  - `enabled = true`
- Print `>>> Default Admin user created: admin / admin123` (stdout).

If an `admin` user is found, it logs `>>> Admin user already exists. Skipping
creation.` and is otherwise idempotent.

This is a development convenience. A production rebuild MUST:

1. Read the bootstrap admin password from a secret (env var or secrets
   manager).
2. Surface a clear startup error if no bootstrap source is configured and the
   users table is empty.
3. Force a password change on first login (not implemented today).

## Configuration values referenced by security

`application.properties` carries:

```
security.remember-me.key=adventure-builder-dev-key
```

Override in production via env var `SECURITY_REMEMBER_ME_KEY` (or the
secrets manager of choice). Rotating this key invalidates all outstanding
remember-me cookies.

## Known gaps

- **Hardcoded admin password.** `DataInitializer.java:30` ships `admin123`
  with a `TODO: CHANGE THIS IN PRODUCTION!`. Production deployments MUST
  inject a secret instead.
- **Hardcoded remember-me key.** `application.properties` ships
  `adventure-builder-dev-key`. Override in production.
- **`UserService.delete` does not block on references.** Deleting a user
  that authors or plays an adventure throws a foreign-key constraint at the
  DB level rather than producing a friendly error. Add explicit checks (or
  cascade) and surface the result via Notification/Dialog.
- **No password-change view.** A user cannot change their own password from
  the UI today. ADMIN can edit other users; users cannot self-service.
- **No account locking / failed-login throttling.** Spring Security's
  `lockable` flags exist on `UserData` but always return `true`. Failed
  logins are logged but not counted.
- **No audit log.** Login events, role assignments, and access decisions are
  not recorded.
- **`AdventureAccessService` admin-only methods rely on caller verification.**
  The `assign*` / `remove*` methods carry `// ADMIN only — callers must
  verify the role` in the source. A rebuild SHOULD enforce this with
  `@PreAuthorize("hasRole('ADMIN')")` rather than trusting callers.
- **No HTTPS enforcement / HSTS configuration.** The app is currently served
  on plain HTTP in development. Production requires terminating TLS at a
  reverse proxy and adding `requiresChannel().anyRequest().requiresSecure()`
  to `SecurityConfig`.
- **Cross-store atomicity gap.** `createAdventure` writes MongoDB before
  the JPA transaction commits — see
  [`05-persistence-and-mappers.md` § Cross-store consistency](05-persistence-and-mappers.md#cross-store-consistency).
